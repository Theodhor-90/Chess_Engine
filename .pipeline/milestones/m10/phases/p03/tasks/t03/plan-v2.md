## 3. Implementation Details

### 3.1 `crates/nnue/src/arch.rs` — `NetworkDims` struct

**Purpose**: Enable runtime-parameterized layer dimensions for loading networks of varying sizes.

**Exports**: `NetworkDims`

**Key logic**:
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct NetworkDims {
    pub halfkp_features: usize,
    pub l1_size: usize,
    pub l2_size: usize,
    pub output_size: usize,
}

impl NetworkDims {
    pub fn default_full() -> Self {
        Self {
            halfkp_features: HALFKP_FEATURES,
            l1_size: L1_SIZE,
            l2_size: L2_SIZE,
            output_size: OUTPUT_SIZE,
        }
    }
}
```

Existing compile-time constants (`HALFKP_FEATURES`, `L1_SIZE`, `L2_SIZE`, `OUTPUT_SIZE`) remain unchanged — they serve as the default/full-network dimensions and are used in contexts where no network is loaded (e.g., `AccumulatorStack` initialization in tests and when `network: None`).

### 3.2 `crates/nnue/src/network.rs` — Dynamic `Network`

**Purpose**: Allow `Network` to hold weights for any valid dimension set.

**Changes**:
- Add `dims: NetworkDims` field to `Network`.
- Change `input_bias` from `Box<[i16; L1_SIZE]>` to `Box<[i16]>`.
- Change `hidden1_bias` from `Box<[i32; L2_SIZE]>` to `Box<[i32]>`.

### 3.4 `crates/nnue/src/loader.rs` — Dynamic loading

**Purpose**: Load networks of any valid dimension set from `.nnue` files.

**Changes**:

## 8. Design Deviations

### Deviation 1: Dynamic dimensions via `NetworkDims` struct instead of compile-time const changes

**Parent spec requires**: "crates/nnue must support parameterized layer sizes" and "the distilled network uses the same inference codepath from Phase P01, differing only in layer dimensions."

**Why compile-time parameterization is problematic**: The current codebase uses compile-time constants (`L1_SIZE`, `L2_SIZE`) throughout `arch.rs`, and these constants are embedded in fixed-size arrays (`[i16; L1_SIZE]`). Parameterizing at compile time would require either (a) recompiling the engine binary for each network size (preventing runtime network loading), or (b) Rust const generics across the entire crate which would massively complicate the type signatures of `Network`, `Accumulator`, `AccumulatorStack`, and `forward()`.

**Alternative chosen**: A `NetworkDims` struct stored in `Network` enables runtime-parameterized dimensions. Fixed-size arrays become heap-allocated `Box<[T]>` or `Vec<T>`. The compile-time constants remain as defaults for the full network. This allows a single binary to load and infer over any dimension set encoded in a `.nnue` file, which is what the task spec actually requires ("the distilled network uses the same inference codepath").

### Deviation 2: HalfKP feature count remains fixed at 40960 across all candidates

**Parent spec states**: Candidates include a "half-width input layer variant" (e.g., 256→128 for L1).

**Clarification**: "Input layer" in NNUE terminology refers to the first hidden layer (L1/accumulator), not the feature input count. HalfKP features are determined by the board representation (64 king squares × 10 piece types × 64 squares = 40960) and cannot be meaningfully halved without changing the feature scheme. The "half-width input" variant halves L1 (the accumulator width from 256 to 128), not the feature count. This is consistent with how NNUE distillation works in practice (e.g., Stockfish's SFNNv1→v2 architecture changes). All four candidates use `halfkp_features = 40960`.

### Deviation 3: `tools/distill` is standalone, not a workspace member

**Parent spec does not address this, but previous plan (v1) added it to workspace members.**

**Why standalone**: `tools/training` — which has the identical structure and identical `bullet_lib` git dependency — is NOT in the workspace `members` list. This is because `bullet_lib` as a git dependency conflicts with workspace dependency resolution. `tools/distill` follows the same convention: standalone crate, built via `cargo build --manifest-path tools/distill/Cargo.toml`, not added to workspace `members`.
