### `tools/training/convert.py`

**Purpose**: Converts raw floating-point weights from the training script's output into the project's custom `.nnue` binary format as defined in `crates/nnue/src/format.rs`.

**Input format** (produced by `tools/training/src/main.rs` save callback):
- 24-byte header: 4 × `u32` dimensions `(40960, 256, 32, 1)` + 2 × `u32` reserved, all little-endian.
- Flat weight arrays in layer order, all as little-endian `f32`:
  1. Input weights: `40960 × 256` floats
  2. Input bias: `256` floats
  3. Hidden1 weights: `32 × 512` floats (512 = 2 × 256, concatenated white/black accumulators)
  4. Hidden1 bias: `32` floats
  5. Hidden2 weights: `32` floats
  6. Hidden2 bias: `1` float

**Output format** (project's `.nnue` as defined in `crates/nnue/src/format.rs`):
- 28-byte header: magic `b"NNUE"` (4 bytes), version `1u32`, architecture hash (computed via `architecture_hash()`), dimensions `(40960u32, 256u32, 32u32, 1u32)` — all little-endian.
- Weight data arrays in order:
  1. Input weights: `40960 × 256` as `i16` (quantized: `round(float × 2^6)`, clamped to `[-32768, 32767]`)
  2. Input bias: `256` as `i16` (quantized: `round(float × 2^6)`, clamped to `[-32768, 32767]`)
  3. Hidden1 weights: `32 × 512` as `i8` (quantized: `round(float × 2^6)`, clamped to `[-128, 127]`)
  4. Hidden1 bias: `32` as `i32` (quantized: `round(float × 2^12)` — product of two `2^6` quantization stages, clamped to `i32` range)
  5. Hidden2 weights: `32` as `i8` (quantized: `round(float × 2^6)`, clamped to `[-128, 127]`)
  6. Hidden2 bias: `1` as `i32` (quantized: `round(float × 2^12)`, clamped to `i32` range)

**Architecture hash computation** (must match `crates/nnue/src/format.rs::architecture_hash()`):
```python
def architecture_hash(dims):
    h = 0
    for d in dims:
        h ^= d
        h = ((h << 7) | (h >> 25)) & 0xFFFFFFFF
    return h
```

**Input validation**: The script reads the 24-byte input header and verifies dimensions match `(40960, 256, 32, 1)` before converting. It also verifies the input file size matches the expected total bytes. On mismatch, exit with error code 1 and a descriptive message.

**CLI interface**:
```
python3 tools/training/convert.py --input data/training-output/epoch_100.bin --output nets/nn.nnue
```

Exit code 0 on success, non-zero with error message on failure.

## 8. Design Deviations

**Deviation 1: Training script and weight conversion script as pipeline tooling**

- **Parent spec states**: "Training pipeline using `bullet` or `nnue-pytorch` produces a trained `.nnue` network file" and "Training configuration files (hyperparameters, data paths, etc.)" under files to create.
- **What we produce**: A Rust training script (`tools/training/src/main.rs` with `tools/training/Cargo.toml`) and a Python conversion script (`tools/training/convert.py`) in addition to documentation.
- **Why this is necessary**: Bullet does not natively output arbitrary custom `.nnue` formats. Bullet provides a save callback mechanism where each engine defines its own weight serialization. The standard approach in the chess engine community is: (1) the engine project writes a bullet training script that defines the architecture and registers a save callback, (2) a separate conversion/quantization tool serializes raw floating-point weights into the engine's custom binary format. This project's `.nnue` format has a project-specific 28-byte header with a custom architecture hash and specific quantization scheme (`i16`/`i8`/`i32` at `2^6`/`2^12` scales) that no external tool produces natively.
- **Spec justification**: The task spec's deliverable "documentation of the training pipeline: exact commands, configuration files, data requirements, and expected outputs" encompasses all tooling needed to execute the pipeline. The training script is the authoritative source of hyperparameters and architecture definition (fulfilling the "training configuration files" deliverable). The conversion script is a pipeline artifact required to produce the `.nnue` deliverable. Both are analogous to build tools.

**Deviation 2: SPRT validation is a joint T02+T03 exit criterion**

- **Parent spec states**: "An SPRT self-play match (Elo0=0, Elo1=30) using CuteChess-CLI passes the H1 hypothesis."
- **Why T02 alone cannot satisfy this**: The SPRT match requires an engine binary capable of using NNUE evaluation during search. The UCI `EvalMode` option and search-level NNUE integration are implemented in T03 (Evaluation Switchover), not T02. Additionally, `tools/sprt` does not currently support per-engine UCI options — it passes engine paths to CuteChess-CLI as `cmd=<path>` without any mechanism to set `option.EvalMode=nnue` on one engine but not the other.
- **Chosen resolution — explicit handoff contract with T03**:
  - **T02 responsibility**: Produce a trained `nets/nn.nnue` file that loads correctly and produces valid inference scores (`|score| < 3000` cp for the starting position). Document the complete SPRT validation procedure with a CuteChess-CLI template command (using `option.EvalMode=nnue`) in `docs/training-pipeline.md`.
  - **T03 responsibility**: Integrate NNUE eval into the search loop, add the `EvalMode` UCI option, finalize the SPRT invocation mechanism (either extend `tools/sprt` with per-engine UCI option flags or invoke CuteChess-CLI directly using T02's template), and execute the SPRT match using `nets/nn.nnue`. If SPRT rejects, T03 reports failure so the T02 pipeline can be iterated following the documented remediation steps.
  - **Joint exit criterion**: The phase-level exit criterion "SPRT self-play match passes" is satisfied when T03 runs the match and it passes. T02's contribution is network quality; T03's contribution is integration correctness.
- **Why not build a temporary harness in T02**: Building a temporary search integration solely for SPRT testing would duplicate T03's scope (search integration, UCI option), violate scope discipline, and produce throwaway code. The handoff contract is cleaner and ensures each task owns its deliverables.

**Deviation 3: Training script kept outside the Cargo workspace**

- **Parent spec implies**: Tools reside in `tools/` and are workspace members (T01 added `tools/datagen` to workspace members).
- **What we do instead**: `tools/training/Cargo.toml` is built standalone with `--manifest-path`, not added to the root `Cargo.toml` workspace members.
- **Why**: `bullet_lib` has a heavy dependency tree including optional CUDA/GPU support. Adding it to the workspace would make `cargo build`, `cargo test`, and `cargo clippy` at the workspace root pull in these dependencies, slowing down CI and development builds for all contributors — even those not training networks. The training script is used infrequently (only during NNUE training iterations) and does not need to participate in the main build.
