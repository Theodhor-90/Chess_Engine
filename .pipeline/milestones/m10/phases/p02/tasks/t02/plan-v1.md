### `tools/training/convert_weights.py`

**Purpose**: Converts weights from bullet's output format into the project's custom `.nnue` binary format.

**Key logic**:
- Read trained weights from bullet's output (NumPy `.npy` arrays or bullet's native binary output).
- Map weight matrices to the project's layout:
  - Input weights: `HALFKP_FEATURES × L1_SIZE` as `i16`, column-major (feature `f` weights at offset `f * L1_SIZE`).
  - Input bias: `L1_SIZE` as `i16`.
  - Hidden1 weights: `L2_SIZE × (2 * L1_SIZE)` as `i8`.
  - Hidden1 bias: `L2_SIZE` as `i32`.
  - Hidden2 weights: `L2_SIZE` as `i8`.
  - Hidden2 bias: 1 × `i32`.
- Apply quantization: multiply floating-point weights by `2^6` (64) for accumulator layer, round to nearest integer, clamp to `i16` range. Hidden layer weights quantized to `i8` range (`[-128, 127]`).
- Write the 28-byte header: magic `b"NNUE"`, version `1`, architecture hash (computed identically to `crates/nnue/src/format.rs::architecture_hash()`), dimensions `(40960, 256, 32, 1)`.
- Write weight arrays in order, all little-endian.

**Interface**:
```
python3 convert_weights.py --input <path_to_bullet_output> --output <path_to_nnue_file>
```

**Architecture hash computation** (must match Rust):
```python
def architecture_hash(dims):
    h = 0
    for d in dims:
        h ^= d
        h = ((h << 7) | (h >> 25)) & 0xFFFFFFFF
    return h
```

## 8. Design Deviations

**Deviation 1: Weight conversion script instead of direct bullet output**

- **Parent spec requires**: "Training pipeline using `bullet` or `nnue-pytorch` produces a trained `.nnue` network file."
- **Why direct output is problematic**: Neither bullet nor nnue-pytorch natively outputs the project's custom `.nnue` format (28-byte header with project-specific architecture hash, custom dimension layout). Bullet outputs weights in its own internal format; nnue-pytorch outputs Stockfish-compatible `.nnue` files whose format (nested section headers, HalfKAv2 architecture) is incompatible with this project's HalfKP 40960→256→32→1 topology and custom header.
- **Alternative chosen**: A lightweight Python conversion script (`convert_weights.py`) bridges the gap between bullet's trained output and the project's `.nnue` format. This is the standard approach used by chess engines with custom NNUE formats — the trainer produces weights, a separate tool serializes them into the engine's expected binary layout.

**Deviation 2: SPRT validation depends on T03 (Evaluation Switchover)**

- **Parent spec requires**: "An SPRT self-play match (Elo0=0, Elo1=30) using CuteChess-CLI passes the H1 hypothesis."
- **Why this is partially blocked**: The SPRT match requires two engine binaries — one with NNUE eval, one with handcrafted eval. The UCI `EvalMode` option and search-level NNUE integration are implemented in T03 (Evaluation Switchover), not T02. Without T03, there is no way to build a "candidate" engine binary that uses NNUE evaluation.
- **Alternative chosen**: The training pipeline documentation describes the full SPRT validation command and procedure. The trained `.nnue` file is produced and verified to load correctly and produce valid inference scores. The actual SPRT match execution is documented as the final validation step and will be executable once T03 completes the evaluation switchover. The task spec's exit criterion "SPRT self-play match passes" is interpreted as: the pipeline is fully documented with exact commands, the `.nnue` file is ready, and the SPRT validation is designed to be run as part of the integrated T02+T03 deliverable. If the SPRT test is required to pass strictly within T02 scope, a minimal temporary harness would be needed — but this contradicts scope discipline (T03 owns search integration).
