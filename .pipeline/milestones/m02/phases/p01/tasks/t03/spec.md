# Task T03 — King Move Generation

## Objective

Implement pseudo-legal king move generation using pre-computed attack tables within the `movegen` crate, excluding squares attacked by the opponent.

## Deliverables

1. **King attack table** — a `[Bitboard; 64]` array where each entry contains the bitboard of all squares a king on that square can reach (up to 8 adjacent squares)
2. **Attack table initialization** — computed at compile time (`const`) or via runtime-once initialization (`OnceLock` / `lazy_static`)
3. **King move generation function** that produces all pseudo-legal king moves for the side to move, filtering out squares attacked by the opponent

## Implementation Details

- For each of the 64 squares, pre-compute the king's attack pattern (up to 8 adjacent squares: N, NE, E, SE, S, SW, W, NW)
- The king attack offsets from square (file, rank) are: (±1, 0), (0, ±1), (±1, ±1), clamped to the board
- The move generation function:
  1. Gets the king's square from `Position::piece_bitboard(Piece::new(side, PieceKind::King))` (exactly one bit set — use `.lsb()` to extract)
  2. Looks up `KING_ATTACKS[king_square.index()]`
  3. Removes squares occupied by friendly pieces (`& !Position::occupied_by(side)`)
  4. Removes squares attacked by the opponent — for each candidate target square, check `Position::is_square_attacked(target, opponent_side)` and exclude if true
  5. For each remaining target: if occupied by opponent → `MoveFlag::CAPTURE`, else → `MoveFlag::QUIET`
  6. Appends `Move::new(from, to, flag)` to the move list

**Important**: This is still pseudo-legal in the sense that discovered checks on the king (from moving the king away from a pin line) may not be fully handled by `is_square_attacked` alone — the king itself may need to be temporarily removed from occupancy when checking attacked squares. Verify this behavior during implementation and handle accordingly.

**Note**: Castling moves (O-O, O-O-O) are NOT generated here. They belong to Phase P03/T01.

## Implementation Constraints

- Pre-computed table must be initialized before first use (prefer `const` if feasible, otherwise `OnceLock`)
- Lives in the `movegen` crate alongside pawn and knight generation from T01/T02
- The opponent attack check uses `Position::is_square_attacked(square, opponent_color)` from the `board` crate
- No `unsafe` unless provably necessary. No `unwrap()` in library code.
- Unit tests colocated via `#[cfg(test)] mod tests`

- `Color`, `Piece`, `PieceKind::King`

## Dependencies on Other Tasks

- **T02** — no direct dependency, but T02 should be complete first to maintain ordering. King generation builds on the same crate structure.

## Verification Criteria

1. `cargo test -p movegen` passes all king-related unit tests
2. King attack table contains correct attack bitboards for all 64 squares
3. Unit tests verify:
   - King on a corner square (e.g., A1) has exactly 3 attack squares in the table
   - King on an edge square (e.g., A4) has exactly 5 attack squares in the table
   - King on a center square (e.g., D4) has exactly 8 attack squares in the table
   - Friendly-occupied squares are excluded from move targets
   - Squares attacked by the opponent are excluded from move targets
   - Captures on opponent-occupied squares use `MoveFlag::CAPTURE`
   - Non-captures use `MoveFlag::QUIET`
4. King cannot move to a square attacked by an opponent piece (verified via test positions)
5. Castling moves are NOT generated by this function
6. `cargo clippy -- -D warnings` passes
7. `cargo fmt --check` passes
