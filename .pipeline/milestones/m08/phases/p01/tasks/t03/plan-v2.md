# T03 — Self-Play Framework: Implementation Plan

## 1. Objective

Build a standalone binary (`tools/selfplay`) that orchestrates self-play games between two instances of the chess engine via UCI. The framework starts each game from a randomized opening position, detects crashes, illegal moves, assertion failures, and time losses, adjudicates game-ending conditions on the harness side, and reports aggregate results. The primary verification target is 2,000+ games without crashes, illegal moves, or assertion failures.

## 2. Architecture Overview

The self-play binary spawns a fresh pair of engine child processes for each game (matching CuteChess-CLI behavior). The harness maintains its own `Position` using the `chess-board` and `chess-movegen` crates to validate every move the engines produce and to detect checkmate, stalemate, and draw conditions. This harness-side adjudication is strictly more robust than relying on engine self-reporting.

```
tools/selfplay/src/main.rs
    │
    ├── CLI parsing (manual env::args)
    ├── Opening book loading (FENs from openings.txt)
    │
    └── Game loop (one iteration per game):
        ├── Spawn two engine processes (white, black)
        ├── UCI handshake (uci → uciok, isready → readyok)
        ├── Move loop:
        │   ├── Send "position fen <fen> moves <history>" to active engine
        │   ├── Send "go movetime <T>" to active engine
        │   ├── Wait for "bestmove <move>" with timeout
        │   ├── Validate move via harness Position + legal move list
        │   ├── Apply move to harness Position
        │   ├── Check for checkmate / stalemate / draw conditions
        │   └── Switch sides
        ├── Send "quit" to both engines, wait for exit
        └── Record game result and any errors
```

## 5. Opening Positions Data

**File**: `tools/selfplay/openings.txt`
**Format**: One FEN string per line (full 6-field FEN). Lines starting with `#` are comments.
**Count**: 200 positions sourced from established opening books — common ECO openings played out 4–8 half-moves from the starting position to provide game diversity. Positions include a mix of open games (1.e4 e5), semi-open games (Sicilian, French, Caro-Kann), closed games (1.d4 d5), Indian systems, and English/Réti openings.

**Source**: Positions generated by replaying standard opening lines to fixed depth. Each FEN represents a position after a short well-known opening sequence (e.g., after 1.e4 e5 2.Nf3 Nc6 → the resulting FEN). This avoids any copyright concerns and ensures positions are sound.

**Selection during play**: Games cycle through openings sequentially. If `games > len(openings)`, the list wraps around. Each opening is played from both sides in consecutive games (game N uses opening N/2 with white, game N+1 uses the same opening with black) to ensure symmetric testing.

### Step 4: Implement `tools/selfplay/src/main.rs`

The implementation is organized into the following logical sections within a single file (no separate modules needed — the total size is ~450–550 lines, consistent with `tools/tactical/src/main.rs` at 555 lines).

#### Section A: CLI Parsing (~40 lines)

Parse `--engine`, `--openings`, `--games`, `--movetime`, `--maxmoves` from `std::env::args()` using an index-advancing loop (same pattern as `tools/tactical`). Print usage and exit(1) on missing required arguments.

#### Section B: Opening Loading (~15 lines)

Read `openings.txt` via `std::fs::read_to_string`. Split by lines, skip empty lines and `#` comments, collect into `Vec<String>`. Panic if zero openings loaded.

#### Section C: Engine Process Management (~50 lines)

**`spawn_engine(path: &str) -> (Child, BufWriter<ChildStdin>, BufReader<ChildStdout>)`**

Spawn a child process with `stdin(Stdio::piped())` and `stdout(Stdio::piped())`. Take ownership of stdin/stdout handles. Perform UCI handshake:
1. Send `uci`, wait for `uciok`
2. Send `isready`, wait for `readyok`

**`send_line(writer: &mut impl Write, line: &str)`** — write line + newline, flush. Identical to tactical tool.

**`wait_for(reader: &mut impl BufRead, expected: &str)`** — read lines until match. Identical to tactical tool.

**`wait_for_bestmove_timeout(reader: &mut impl BufRead, timeout: Duration) -> Result<String, TimeoutError>`**

Unlike `wait_for_bestmove` in the tactical tool, this version enforces a timeout. Implementation: set the child stdout to non-blocking is not portable, so instead use a dedicated reader thread per engine:

- Before the game loop, spawn a reader thread that continuously reads lines from engine stdout into a `mpsc::channel`.
- In the main thread, call `receiver.recv_timeout(timeout)` to get lines.
- This cleanly separates blocking I/O from timeout enforcement.
- Timeout value: `movetime * 10` (give the engine 10× its allocated think time before declaring a time loss, to account for process overhead).

#### Section D: Move Validation (~40 lines)

**`validate_move(pos: &mut Position, move_str: &str) -> Option<chess_types::Move>`**

1. Generate legal moves via `generate_legal_moves(pos)`
2. Format each legal move as UCI string (using `Display` for `Move`)
3. Find the one matching `move_str`
4. Return `Some(move)` if found, `None` if the move is illegal

This uses the same crates (`chess-board`, `chess-movegen`, `chess-types`) as the tactical tool.

#### Section E: Game Adjudication (~80 lines)

After each move is applied to the harness `Position`:

**Checkmate**: `generate_legal_moves(pos)` returns empty AND king is in check → side to move is checkmated.

**Stalemate**: `generate_legal_moves(pos)` returns empty AND king is NOT in check → draw.

**Fifty-move rule**: `pos.halfmove_clock() >= 100` (100 half-moves = 50 full moves) → draw.

**Threefold repetition**: Maintain a `Vec<u64>` of Zobrist hashes for the current game. After each move, count occurrences of the current hash. If count ≥ 3 → draw.

**Insufficient material**: Check if only kings remain, or K vs K+B, or K vs K+N. Specifically:
- Total piece count (excluding kings) ≤ 1, and that piece (if any) is a bishop or knight → draw.
- Both sides have exactly one bishop and the bishops are on same-colored squares → draw.

**Max moves**: If ply count reaches `max_moves` (default 300) → draw (prevents infinite games).

King-in-check detection uses `pos.is_square_attacked(king_square, opponent_color)`.

#### Section F: Game Loop (~100 lines)

```
for game_num in 0..args.games:
    opening_index = game_num / 2  (mod openings.len())
    opening_fen = openings[opening_index]

    pos = Position::from_fen(opening_fen)
    hash_history = vec![pos.hash()]
    move_history: Vec<String> = vec![]
    ply_count = 0

    Spawn two engine processes (white_engine, black_engine)
    Start reader threads for both engines' stdout

    loop:
        active_engine = if pos.side_to_move() == White { &white } else { &black }
        side_name = if pos.side_to_move() == White { "white" } else { "black" }

        Send "isready", wait for "readyok"
        Send "position fen <opening_fen> moves <move_history joined by spaces>"
        Send "go movetime <movetime>"

        match wait_for_bestmove_timeout(timeout):
            Err(timeout) →
                Record TimeLoss for side_name, break
            Ok(bestmove_str) →
                if bestmove_str == "0000" or "(none)":
                    // Engine resigned or has no move — treat as loss
                    break

        match validate_move(&mut pos, &bestmove_str):
            None →
                Record IllegalMove for side_name, break
            Some(mv) →
                undo = pos.make_move(mv)
                move_history.push(bestmove_str)
                ply_count += 1
                hash_history.push(pos.hash())

        Check adjudication conditions (checkmate, stalemate, draws)
        If game over → record result, break

    Send "quit" to both engines
    Wait for both child processes to exit
    Check exit codes: non-zero → record as assertion failure (crash)

    Print one-line game summary to stdout
```

**Exit code**: 0 if zero crashes + zero assertion failures + zero illegal moves. 1 otherwise. Time losses are reported but do not cause a non-zero exit (a few time losses under very fast time controls are acceptable per the task spec).

#### Section H: Error Detection Details

| Error type | Detection mechanism |
|---|---|
| **Crash** | Engine stdout returns EOF (reader thread channel closes), or `child.wait()` returns before game ends |
| **Assertion failure** | `child.wait()` returns non-zero exit code (subset of crash) |
| **Illegal move** | Engine's `bestmove` does not match any legal move in the harness `Position` |
| **Time loss** | `recv_timeout` exceeds the timeout threshold (`movetime * 10`) |

When any error is detected mid-game, the game is immediately terminated. Both engines receive `quit` (if still running) and the error is recorded. The framework continues to the next game — it does not abort the entire run on a single error.

## 8. Design Deviations

**Deviation 1: Fresh engine processes per game instead of persistent reuse**

- **Parent spec states**: "The self-play harness launches two instances of the engine binary and communicates via UCI stdin/stdout."
- **Issue**: The phrasing could imply two long-lived engine instances across all 2,000 games. However, reusing processes risks state leakage between games (transposition table, history tables) that could mask bugs. Additionally, crash detection is simpler with per-game spawning — if a process dies mid-game, we know exactly which game caused it.
- **Alternative chosen**: Spawn a fresh pair of engine processes for each game. Send `uci`/`uciok`, `isready`/`readyok` at the start, `quit` at the end. This is how CuteChess-CLI operates by default and is the most robust approach for detecting assertion failures and crashes per-game.

**Deviation 2: Harness-side game adjudication instead of relying on engine output**

- **Parent spec states**: "Detect crashes, illegal moves, assertion failures, and time losses."
- **Clarification**: The spec does not prescribe where game-ending conditions (checkmate, stalemate, draws) are detected. Detecting them on the harness side (by maintaining a parallel `Position` and generating legal moves) is strictly more robust than relying on engine self-reporting, because it catches bugs where the engine fails to recognize checkmate or claims a draw incorrectly. This is the same approach used by CuteChess-CLI and other tournament managers.

**Deviation 3: Timeout via reader thread + mpsc channel instead of non-blocking I/O**

- **Parent spec states**: Time losses should be detected when the engine exceeds allocated time.
- **Issue**: Rust's `BufReader::read_line` is blocking and there is no portable way to set non-blocking mode on a child process pipe without platform-specific code or external crates. Using a reader thread with `mpsc::recv_timeout` achieves the same result using only standard library facilities, consistent with the project's convention of minimizing external dependencies.
